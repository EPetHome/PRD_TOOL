# 模块 3 · AI 构建包 · 产出物规范

> 版本:V1.0
> 性质:定义"AI 读的那份文件"应该长什么样 —— PM 不需要手填本文件,由模块 2 文档(尤其第 8 章下游转译输入) + 模块 3 转换 Prompt 自动产出一份《AI 构建包》
> 受众(读本文件):PM / 开发者 / 审核者
> 受众(读按本规范产出的"AI 构建包"):AI 工具(Cursor / Claude Code / v0 / Lovable / 通用大模型)

---

## 一、核心原则

1. **字段全局唯一** · 整份 AI 构建包中,同一概念只能有一个命名;一旦命名,其他位置引用该命名
2. **无模糊词** · 禁用"大约 / 可能 / 适当 / 一些 / 几个 / 视情况"等模糊词,全部替换为具体值或显式说"待产品方补充"
3. **无重复叙述** · 同一信息只在一处给出定义,其他位置引用;不允许同一规则在两处分别表述
4. **执行顺序明确** · 章节顺序就是 AI 构建顺序(先数据模型 → 再页面 → 再交互 → 最后合规守门),AI 按顺序读就能按顺序建
5. **禁区前置** · 任何"AI 不得做"的事放在对应章节开头,不在末尾补充
6. **来源可追溯** · 模块 2 不手填数据库字段表;AI 构建包中的工程字段名由 AI 从模块 2 的用户可见输入 / 输出 / 状态口径规范化生成,并在《自检结论》中说明来源映射

---

## 二、AI 构建包的骨架(9 章 · 顺序严格)

AI 产出的文件是一份 Markdown,包含以下 9 个章节(编号 0-8),顺序不可调。

### 0 · 元数据

```markdown
# [[ 产品名 ]] · AI 构建包

| 项 | 值 |
|---|---|
| 产品 ID | [[ 短横线分隔的英文或拼音,全局唯一 ]] |
| 版本 | [[ 与模块 2 版本号对齐 ]] |
| 来源模块 2 | [[ 文件路径 ]] |
| 生成时间 | [[ YYYY-MM-DD ]] |
| 面向 AI | [[ Cursor / Claude Code / v0 / Lovable / 通用 ]] |
```

### 1 · 技术选型

```markdown
## 1. 技术选型

- **前端框架**:[[ 具体框架 + 版本,如 "React 18 + TypeScript" 或 "单文件 HTML + Tailwind CDN" ]]
- **状态管理**:[[ 具体方案或"无" ]]
- **样式方案**:[[ 具体方案 ]]
- **后端存储**:[[ localStorage / IndexedDB / 文件 / 数据库 / 无(纯前端) ]]
- **部署形态**:[[ 单文件 HTML / 可部署前端 / 全栈 / 其他 ]]
- **AI 调用方**:[[ 浏览器直接调 / 后端代理调 / 无 AI 调用 ]]
```

**来源优先级**:优先读取模块 2 第 8.5"技术选型最小口径";第 8.5 未填写时,再读取第 5 章技术路线叙事;仍无法判断时写 `[待产品方补充]`。

### 2 · 数据模型

```markdown
## 2. 数据模型

### 实体 · [[ 实体名,PascalCase,全局唯一 ]]

| 字段名 | 类型 | 必填 | 取值约束 | 说明 |
|---|---|---|---|---|
| [[ camelCase 唯一名 ]] | [[ string / number / boolean / date / enum / array / object ]] | [[ 是 / 否 ]] | [[ 具体范围或枚举值,禁用"合理""适当" ]] | [[ 一句话 ]] |

(每个实体一张表,按重要性排序)
```

**生成口径** · 优先从模块 2 第 8.2"用户可见输入项"、第 8.3"用户可见输出结果"、第 8.4"关键状态语义"抽取实体和字段;模块 2 没有写工程字段名时,AI 按 camelCase 生成,但不得新增模块 2 没提到的业务概念。

**禁区** · 本章节不写业务叙事、不写 UI 描述、不写决策逻辑。

### 3 · 页面结构

```markdown
## 3. 页面结构

### 页面 · [[ 页面名,全局唯一 ]]

- **路由**:[[ 如 "/" / "/detail/:id" / "无(单页)" ]]
- **读取的实体**:[[ 实体名列表 ]]
- **写入的实体**:[[ 实体名列表 ]]
- **主要区域**:
  - [[ 区域名 ]] · [[ 一句话说明作用 ]]
  - [[ 区域名 ]] · [[ 一句话说明作用 ]]
- **用户身份限制**:[[ 如 "游客可访问" / "仅登录后" / "仅管理员" / "无限制" ]]
```

**禁区** · 本章节不写 CSS 样式细节、不写组件库选型、不写动画时长。那些在前端实现时自行决定。

### 4 · 交互流程

```markdown
## 4. 交互流程

### 流程 · [[ 流程名,全局唯一 ]]

**前置条件**:[[ 用户当前在什么页面 / 什么状态 ]]

| 步骤 | 触发 | 系统动作 | 涉及实体 | 失败态处理 |
|---|---|---|---|---|
| 1 | [[ 用户点击 / 输入 / 其他 ]] | [[ 具体到字段级,如"将 User.name 更新为输入值" ]] | [[ 实体名 ]] | [[ 具体错误态,如"弹出 '名称不能为空'" ]] |

**终点**:[[ 流程正常结束时用户看到什么 ]]
**禁区**:[[ 此流程中 AI **不得**做的事,如"不得自动删除数据""不得跳过 Step 3" ]]
```

### 5 · AI 决策点(仅当产品包含 AI 能力)

```markdown
## 5. AI 决策点

### 决策点 · [[ 决策点名,全局唯一 ]]

- **触发**:[[ 什么时候调 AI ]]
- **输入**:[[ 字段级别的输入清单 ]]
- **输出**:[[ 字段级别的输出清单 + 每字段约束 ]]
- **模型选择**:[[ 具体模型名 / 供应商 / 或 "由 AI 构建方选择" ]]
- **温度 / 参数**:[[ 具体值或"默认" ]]
- **Prompt 骨架**:用 4 个反引号代码块包裹实际 prompt 文本(含变量占位符)
- **置信度分段策略**:
  - 高(> [[ 值 ]]):[[ 具体处置 ]]
  - 中([[ 值 ]]-[[ 值 ]]):[[ 具体处置 ]]
  - 低(< [[ 值 ]]):[[ 具体处置 ]]
- **失败回退**:[[ AI 调用失败时的回退行为 ]]
- **禁区**:[[ AI 不得输出的内容,如"不得输出医疗诊断""不得泄露其他用户数据" ]]
```

### 6 · 规则判定点(纯规则,不涉及 AI)

```markdown
## 6. 规则判定点

### 规则 · [[ 规则名,全局唯一 ]]

- **触发**:[[ 什么时候执行 ]]
- **输入**:[[ 字段级 ]]
- **判定逻辑**:[[ 伪代码或显式判断条件,禁用模糊词 ]]
- **输出**:[[ 具体值 / 跳转 / 拦截 ]]
- **优先级**:[[ 当多规则同时触发时的顺序,整数 ]]
```

### 7 · 合规红线(必守,最高优先级)

```markdown
## 7. 合规红线

本章所有条目是**硬规则**,AI 构建产品时必须在对应位置植入对应守门逻辑。

### 红线 · [[ 红线名 ]]

- **触发**:[[ 什么输入 / 什么场景触发 ]]
- **必须做**:[[ 具体动作,如"拦截""记录日志""转人工" ]]
- **植入位置**:[[ 第几个流程的第几步 ]]
- **绝对禁止**:[[ AI 不得绕过的事 ]]
```

### 8 · 执行顺序与依赖

```markdown
## 8. 执行顺序与依赖

AI 构建产品时,按以下顺序执行:

1. 建立**数据模型**(第 2 章所有实体)
2. 建立**页面骨架**(第 3 章所有页面的路由和主区域)
3. 实现**交互流程**(第 4 章)
4. 接入 **AI 决策点**(第 5 章,若有)
5. 植入**规则判定**(第 6 章)
6. 植入**合规红线**(第 7 章,覆盖所有流程)
7. 自检:每个字段名唯一、每个禁区已守、每条合规红线已植入
```

## 三、字段命名规范

整份 AI 构建包统一遵守:

- **实体 / 组件名**:PascalCase(如 `User` / `OrderItem` / `ChatMessage`)
- **字段 / 变量名**:camelCase(如 `userName` / `isActive` / `createdAt`)
- **路由 / 文件名**:kebab-case(如 `/user-profile` / `order-detail.tsx`)
- **同一个概念全局只能命名一次**。如果模块 2 中出现多个叫法(如"用户"/"客户"/"使用者"),AI 构建包必须统一为一个命名,并在《自检结论》中列出"命名统一表"

---

## 四、模糊词禁区(整文扫描需全零命中)

禁止出现以下词:

```
大约 · 可能 · 适当 · 一些 · 几个 · 视情况 · 看情况
待后续决定 · 等等 · 差不多 · 若干 · 或许 · 可能会
```

遇到模块 2 原文含以上词的,AI 必须:

- 尝试从上下文推断具体值
- 推断不出来的,写 `[待产品方补充]`,并在《自检结论》末尾列出所有"待补充项"

---

## 五、去重规则

- 同一数据字段在不同章节出现时,**只在第 2 章定义**,其他章节引用字段名
- 同一规则在不同流程触发时,**只在第 6 章定义**,流程中只引用规则名
- 如果模块 2 中一个逻辑在多处叙述,**AI 合并为一条,引用一次**

---

## 六、AI 必须产出的《自检结论》

AI 产出 AI 构建包后,在文件末尾必须产出一段《自检结论》,包含:

1. **命名统一表**(模块 2 原叫法 → 统一命名)
2. **字段唯一性扫描结果**(若有重复,列出;否则写"全局唯一")
3. **模糊词扫描结果**(必须零命中,列出命中项或写"零命中")
4. **合规红线植入位置列表**(每条合规红线 → 植入到第几流程第几步)
5. **待产品方补充项清单**(AI 判断不出的具体值,交给 PM 决策)
6. **来源映射表**(模块 2 章节 / 原始用户可见名词 → AI 构建包中的实体 / 字段 / 页面 / 流程)

---

## 七、输出格式规范

- **格式**:Markdown,章节编号 0-8,章节顺序不可调
- **表格语法**:标准 Markdown 表格
- **Prompt 骨架**:用 4 个反引号代码块包裹(避免与 Markdown 表格和 3 反引号冲突)
- **整份产出物必须可被另一个 AI 直接读取执行**,不依赖人类解读

---

## 八、产出物的文件命名与位置

- **文件名**:`[[ 产品短名 ]]-AI 构建包.md`
- **存放位置**:与模块 2 文档同目录
- **版本号**:与模块 2 文档对齐

---

## 九、本规范的边界

- 本规范**不规定** AI 构建时具体用什么 UI 框架 / 组件库 —— 那由第 1 章"技术选型"逐产品决定
- 本规范**不规定**设计视觉样式 —— 那是模块 1 原型的责任
- 本规范**不规定**人类读这份构建包能否读懂 —— 可读性不是首要目标,**AI 可执行才是**
