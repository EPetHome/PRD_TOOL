# 模块 1 · 原型脚手架

> 版本:V1.0
> 受众:PM(不会前端,用 AI 工具做可交互原型)
> 目的:PM 填完模块 2 后,复制本文件内的 Prompt 骨架 + 模块 2 文档 → 喂给任意 AI → 得到一份**可直接双击打开**的单文件 HTML 原型,交给算法 / 前端 / 后端

---

## 一、模块 1 是什么(2 分钟看完)

**模块 1 的产出物 = 一份可以双击打开就能点击的 HTML 原型文件。**

它**不是**:

- ❌ Figma 原型 —— Figma 要软件、要账号、导出不顺手,新同事看起来不方便
- ❌ 线框图 —— 线框图不能点、不能看异常态、不能看状态流转
- ❌ 可上线的产品 —— 原型只是"让工程师看到长什么样",不是可交付代码

它**是**:

- ✅ 一份 `.html` 文件,双击用浏览器打开就能用
- ✅ 能点击核心按钮、能看到核心流程从头走到尾
- ✅ 能切换"正常态 / 失败态 / 空态 / 加载态"四种界面
- ✅ 用 Tailwind CDN,不依赖任何工具链(不用 npm、不用跑本地服务)

**一句话:模块 1 是"能双击打开、会动的 PRD"。**

---

## 二、工程师要从原型里得到什么

| 工程师角色 | 看原型是为了 | 具体关注哪些地方 |
|---|---|---|
| **前端** | 知道要做哪些页面、页面长什么样、点击后怎么跳 | 所有页面的整体布局、页面间的跳转、各组件的基本形态 |
| **后端** | 知道要准备哪些数据、哪些接口、参数从哪里来 | 原型里的参数配置入口、表单字段、跨页面传递的数据 |
| **算法** | 知道 AI 在哪些节点被调用、输入输出长什么样 | 原型里标记为"🤖 AI 决策点"的节点、AI 返回结果的展示位置 |

**三类人都关心的:异常态**。原型必须展示"失败了怎么显示""空数据怎么显示""加载中怎么显示",否则工程师会按自己的理解瞎做,最后和 PM 预期不一致。

## 三、整体布局规范(强制 · 决定原型的"样子")

所有按本脚手架产出的原型,**必须**遵守以下布局:

### 参考效果

目标效果参考「体质膳调 V2 交互原型控制台」:左侧是一张可操作的控制台卡片,右侧是一台高保真手机壳里的 App 原型。**只学习这种视觉形态和联动方式,不复制它的 iframe / postMessage 实现。**

### 桌面演示台

- **整体背景**:暖灰 / 米白色全屏背景,内容水平居中,形成"演示台"效果,不要做成普通后台页面
- **左侧控制台卡片**(宽 280-320px)· 白色 / 半透明白底,大圆角,柔和阴影,内部是切换器 + 说明卡片
- **右侧手机模型**(居中,iPhone 尺寸 390×844)· 黑色厚边框 + 刘海 / 状态栏 + 屏幕内容滚动区
- **切换协议**:原生 JS 状态对象 + DOM 重渲染。**不用 iframe,不用 postMessage**。

### 控制台必须像"演示控制台",不是普通表单

- 顶部:产品名 + "交互原型控制台"标题 + 一句话说明
- 第一组:产品最核心的业务变量,优先用 3 列 chip 网格展示(如体质 / 渠道 / 商品类目 / 客户等级 / 风险等级)
- 第二组:用户档案 / 用户角色,用下拉或按钮组
- 第三组:关键业务状态 / 异常态,用下拉、Tab 或 checkbox
- 底部:主按钮(如"应用到原型") + "控制台说明"小卡片
- 每次切换或点击主按钮,右侧手机内容必须出现肉眼可见变化
- 控制台**不是主导航**。它只负责切换演示变量,不要把所有页面跳转都交给控制台下拉

### 手机屏必须能自己走流程

- 右侧手机里的卡片、列表、Tab、返回箭头、主按钮都要能点击
- 主流程页面之间的跳转必须发生在手机屏内,不能依赖左侧控制台才能进入下一页
- 控制台最多提供"重置到首页 / 快速定位某一演示页"这类辅助能力,不能替代手机内真实交互
- 所有看起来可点击的元素,必须绑定动作:跳转页面 / 打开弹层 / 切换状态 / 显示 Toast 四选一

### 为什么这么定

- 统一视觉规范 → 团队任何同事打开原型都认得出"这是 PM 套件做的"
- 左控制台 + 右手机模型 → 工程师一眼看清"产品是手机 App,状态可切换"
- 单文件 + DOM 切换 → AI 一次吐出成功率最高,新手不怕跑不出

## 四、原型必须包含的 4 块结构(缺一不可)

### 1. 核心流程主线

从"用户进入产品"到"用户拿到核心价值"的**一条可点击的完整路径**。

举例:
- SaaS 工具:登录 → 首页 → 新建任务 → 查看结果 → 导出
- AI 助手:打开 → 输入问题 → AI 返回 → 用户操作结果
- 电商:浏览 → 加购 → 结算 → 支付 → 订单

流程中每一步都要能在**右侧手机屏内真的点过去**,不能是"贴图 + 文字说明",也不能只能靠左侧控制台切页。

### 2. 四态展示(必含)

每条核心流程的关键步骤要支持以下 4 种状态切换:

| 状态 | 什么时候出现 | 原型要显示 |
|---|---|---|
| **正常态** | 数据正常返回 | 业务主内容 + 用户下一步操作 |
| **失败态** | 用户操作后系统报错 | 错误提示的具体文案 + 用户下一步能做什么 |
| **空态** | 无数据(首次使用 / 数据被清空) | 引导用户去创建 / 导入的入口 |
| **加载态** | 正在等待响应 | 骨架屏 / Loading 动画 / 进度提示 |

在左侧控制台里放一个"状态切换器",让查看者能**切换**这些态,不用复杂操作。

### 3. 参数配置入口

产品所需的**输入字段 / 配置项**要以表单或面板形式出现在原型里。

举例:
- AI 助手要读用户"意图",原型就要有"意图选择"或"自由输入框"
- 产品要按"地区"推送,原型就要有"地区选择器"
- 产品依赖"用户画像",原型就要有"画像切换器"

不要求写真实逻辑,能填 / 能选 / 能切换即可 —— 工程师看到这些入口就知道"哦,后端要给我这些字段"。

### 4. 状态流转示意(产品有状态机时必含)

如果产品有"状态机"(订单状态 / 工单状态 / 审核状态等),原型里要有**一块区域**画出状态之间的流转关系。

可以是:
- 一张 SVG 状态图(3-8 个节点 + 箭头)
- 一个时间轴(已创建 → 审核中 → 已通过 → 已发布)
- 一个徽章列表(展示产品内所有可能的状态)

**不要求动画**,要求"工程师一眼看出一共有几种状态、之间怎么转"。

如果产品没有状态机(如纯搜索工具),本块可省略。

### 补充要求 · 可点击覆盖

模块 2 中规划出的 P0 主流程、核心功能入口、列表项、用户/任务/订单/报告卡片、提交/确认/申请/查看等动作,只要在原型里做成可点击样式,就必须有对应承接。若模块 2 没有显式 P0/P1 标记,默认第 3.3 核心 Journey 和第 4.3 核心功能都按 P0 处理。

- 能展开完整体验的 → 进入对应页面或弹层
- MVP 只占位的 → 给出明确 Toast 或占位页,说明"当前版本暂不展开"
- 涉及状态的 → 点击后状态要变化,并能在页面上看出来

**禁止**出现"按钮看起来能点,但点了没反应"的原型。

## 五、必含 · 演示控制台

在原型左侧放一个"演示控制台卡片",让**查看者(工程师 / 评审者)能快速切换**以下维度:

- 切换产品最核心的业务变量(例如体质 / 商品类目 / 渠道 / 风险等级 / 任务类型)
- 切换用户角色或用户档案(匿名 / 普通用户 / VIP / 管理员 / 典型画像)
- 切换数据场景(空数据 / 少量 / 大量 / 边界)
- 切换四态(正常 / 失败 / 加载 / 空)
- 切换关键业务状态(例如授权等级 / 审核状态 / 消息状态 / 任务状态)

**为什么必含**:产品场景多样时,查看者要手动操作 20 次才能看完所有分支,多数分支会被漏看。加上控制台后 3 分钟能看完主要分支。

产品只有 1-2 个页面、不涉及多用户 / 多状态时,也要保留控制台卡片,但可以只保留"业务变量 / 异常态 / 控制台说明"三组。

## 六、使用步骤(3 步 · 10 分钟)

### 第 1 步 · 准备输入

打开两份文件:

- **填好的模块 2 文档**(`[[ 产品名 ]]-模块 2-文档.md`)
- **本文件**(就是你现在看的这份)

### 第 2 步 · 喂给 AI

把**本文件第七节的"Prompt 骨架"整段复制**,贴到任意 AI 工具(Claude / ChatGPT / Cursor / Claude Code / v0 / Lovable),然后**在 Prompt 后紧接着贴模块 2 文档全文**,发送。

### 第 3 步 · 核对并保存

AI 产出一个 HTML 代码块。把代码块**另存为** `[[ 产品名 ]]-原型.html`,**双击**用浏览器打开,按第八节的核对清单核一遍。核对通过即交付。

## 七、Prompt 骨架(可直接复制)

```
---PROMPT-START---

# 角色

你是一个"PRD → 可点击 HTML 原型"转换器。你的任务是读懂我提供的模块 2 文档(面向人类阅读的产品 PRD),输出一份**单文件 HTML**,让算法 / 前端 / 后端工程师双击打开就能看清这个产品长什么样、怎么点、异常态什么样。

你必须同时扮演两个角色:

## 角色 A:AI 产品经理

- 从模块 2 中抽取核心用户、核心 Journey、核心功能和业务主变量
- 判断哪些 AI 决策点、规则判定点、合规红线必须在原型中显性展示
- 确保主流程完整走通,异常态和边界状态能被左侧控制台快速演示
- 不新增模块 2 没有提到的业务功能,不替 PM 重新设计产品范围

## 角色 B:专业交互设计师

- 规划移动端信息层级:首屏先看到核心价值,再看到输入、结果、解释和下一步
- 明确主操作、次操作、危险操作的视觉优先级,避免所有按钮长得一样
- 为正常 / 失败 / 空 / 加载四态分别设计清晰反馈和下一步动作
- 保证表单、列表、卡片、结果页符合手机端操作习惯,不做 PC 页面缩小版
- 控制页面密度,让工程师一眼看懂结构,让非技术评审者能顺着页面理解产品

# 必须遵守的整体布局(强制)

输出必须是一个"交互原型演示台",视觉目标参考「体质膳调 V2 交互原型控制台」:左侧浮动控制台卡片,右侧高保真 iPhone 手机模型。**只学习这种布局和联动效果,不要复制 iframe / postMessage 实现。**

## 整体舞台

- 页面背景:暖灰 / 米白色(`#f6f4ef` 或相近色),不要做纯白后台页
- 主容器:全屏高度,水平居中,左控制台 + 右手机模型并排,间距 28-48px
- 视觉质感:白色卡片、柔和阴影、圆角、轻边框,整体像产品 Demo 台,不是线框图
- 桌面优先:宽屏下左右并排;窄屏下可上下堆叠,但手机仍保持 iPhone 形态

## 左侧:演示控制台卡片(宽 280-320px)

- 控制台本身必须是一张白色 / 半透明白色浮动卡片,不是满高灰色侧栏
- 顶部:产品名 + "交互原型控制台"标题 + 一句话说明
- 中部:若干切换器(按下文"控制台必含维度")
- 底部:一个主按钮(如"应用到原型") + "控制台说明"小卡片
- 控件风格:业务主变量优先用 3 列 chip 网格;用户档案 / 业务状态可用下拉;四态用 Tab 或按钮组
- 所有控件必须有选中态,选中后样式明显变化
- 控制台只负责切换**演示变量**,不要把它做成页面导航菜单。手机内页面跳转必须靠右侧原型自己的点击完成

## 右侧:iPhone 手机模型(居中显示)

- 手机外壳尺寸接近真实 iPhone,屏幕内容区域按 **390 × 844 px** 设计
- 机身:黑色边框 8px 左右,圆角 40px 以上,柔和大阴影
- 顶部:黑色刘海 / 灵动岛 + 状态栏,状态栏必须含 **9:41** + 信号 / WiFi / 电池
- 屏幕内容:是一个完整 App 页面,不是文档卡片,不是 PC 页面
- 屏幕内部可纵向滚动,底部可有 Home indicator
- 每个页面顶部:返回箭头 + 当前页标题

## 切换协议(强制)

- 左侧切换器绑 click / change 事件;也可以提供"应用到原型"主按钮
- 原生 JS 更新全局状态对象:`window.state = { route, routeStack, primaryVariant, role, selectedUser, businessStatus, errState, dataScene, ... }`
- 状态变化触发 `renderPhone()` 或 `render()` 重绘手机屏
- 每次切换后,右侧手机内容必须有肉眼可见的变化(标题、卡片、结果、提示、状态至少一处变化)
- **不用 iframe、不用 postMessage**,所有内容在同一 DOM 内

## 手机内真实交互协议(强制)

- 右侧手机里的主卡片、列表项、Tab、返回箭头、主按钮、次按钮都必须绑定点击事件
- 页面跳转用 `navigate(route, params)`;返回用 `goBack()`;临时反馈用 `showToast(text)`;确认/表单用 `openModal(type, payload)`
- `routeStack` 记录访问路径,返回箭头必须真的回到上一页,不能只是摆设
- 主流程必须能不碰左侧控制台,只在右侧手机里从首页一路点到流程终点
- 左侧控制台可以改变变量并重渲染当前页面,但不能成为唯一的页面入口
- 所有看起来可点击的元素,必须至少有一种结果:跳转页面 / 打开弹层 / 切换状态 / 显示 Toast。不得存在无响应按钮

# 控制台必含维度

硬性(必含):

- **业务主变量**:优先从模块 2 第 8.1 抽取;若未填写,再从模块 2 全文抽一个最能体现产品差异的变量,用 chip 网格展示。例:体质 / 商品类目 / 渠道 / 风险等级 / 任务类型 / 客群。至少 3 个选项;如果模块 2 不足 3 个,按模块 2 实际数量展示
- **用户角色 / 用户档案**:按模块 2 画像列,至少 2 个(按钮组或下拉)
- **关键业务状态**:优先从模块 2 第 8.4 抽取。例:授权等级 / 审核状态 / 消息状态 / 任务状态 / 订单状态
- **异常态**:正常 / 失败 / 空 / 加载 **四态 Tab 切换**

可选(模块 2 提到则加):

- **数据场景**:空 / 少量 / 大量
- **演示定位**:可提供"重置到首页 / 跳到关键页面"的辅助下拉,但它不能替代手机内点击导航

# 演示控制台的覆盖底线

不要求覆盖所有组合(几十种太多),但**主流程每一步必须至少支持"正常/失败/空/加载"四态切换**,以及**至少 1 个业务主变量 + 至少 2 个画像角色切换**。其余组合降级为"该组合暂不演示"文案,不得留空白页。

# 手机屏内必含 4 类内容(缺一不可)

1. **主流程页面**(3-6 页,抽自模块 2 核心 Journey,通过手机内点击导航进入,不是只靠控制台切换)
2. **异常态页面**(每关键主流程页支持四态切换)
3. **参数配置入口**(表单 / 下拉 / 勾选,优先从模块 2 第 8.2"用户可见输入项"抽取;若缺失,再从第 4.3 核心功能叙事抽取)
4. **状态流转示意图**(若模块 2 有状态机,单辟一页画;无则省)

# 点击覆盖底线(强制)

- 先从模块 2 的 3.3 核心 Journey、4.2 功能清单、4.3 核心功能详述、8.2 输入项、8.3 输出结果中整理"可点击对象清单"
- P0 可点击对象必须做成真实交互:至少进入一个页面 / 弹层 / 状态变化 / Toast
- P0 包括:核心 Journey 中出现的页面入口、列表项、主按钮、用户档案、消息会话、结果/报告、提交/确认动作
- P1 或未来规划功能可以不展开完整页面,但必须显示明确反馈,如"该能力为 V1.1 预留",不能无响应
- 页面内所有 `button`、`a`、卡片、列表项、Tab 如果看起来可点,就必须绑定事件
- 产出 HTML 前自检:搜索所有 `<button`、`onclick`、可点击 class,确认没有死按钮

# 高保真最低标准

- 手机屏首屏必须像真实 App:有导航栏、核心 Hero / 功能卡片 / 列表或表单,不能只有文字说明
- 页面内容必须使用模块 2 的业务内容填充,不要放"示例标题 / 示例内容"
- 正常态要有完整业务结果;失败态、空态、加载态都要有专门视觉,不能只换一行文案

# 必须遵守的技术约束

4. **可运行**:任何人双击 .html 就能用,无需启动本地服务器
5. **字体与样式**:中文默认用系统字体,色调简洁(白底 + 1-2 个主色),不做复杂动效
6. **禁 iframe、禁 postMessage** · 所有内容在同一 HTML 的同一 DOM 内
```

## 九、常见坑与应对

### 坑 1 · AI 产出多文件(.html / .css / .js 分开)

追一句:"请把所有内容合并为一个 .html 文件,CSS 用 Tailwind CDN,JS 内联,不要分文件。"

### 坑 2 · AI 编造了模块 2 没提的功能

追一句:"模块 2 只提到了以下 N 个核心功能,请只保留这些,不要添加新功能。"

### 坑 3 · AI 没做异常态

追一句:"按 Prompt 要求,每个关键页面必须有失败态 / 空态 / 加载态,请补全。"

### 坑 4 · 原型里的文案和模块 2 对不上

追一句:"所有用户可见按钮 / 文案 / 业务名词 / 状态语义必须与模块 2 一致,不得修改。数据库字段名不需要出现在原型里。"

### 坑 5 · 原型太简陋,工程师看不出字段有哪些

回去把模块 2 里的"核心功能详述"里的输入项列得更细,重跑 Prompt。

### 坑 6 · AI 做成普通 Dashboard,不像手机 App 原型

追一句:"请按脚手架里的高保真演示台要求重做:左侧浮动控制台卡片,右侧 iPhone 手机模型,手机内是 390×844 App 页面,不要做 PC Dashboard 或线框图。"

### 坑 7 · 功能都有,但交互不好用

追一句:"请同时按 AI 产品经理和专业交互设计师视角重构原型:保留现有业务范围,重新梳理移动端信息层级、主操作、次操作、四态反馈和每个状态的下一步动作。"

### 坑 8 · 只能靠控制台切页,手机里点不动

追一句:"请把左侧控制台降级为变量调试器,不要作为主导航。右侧手机里的卡片、列表、Tab、返回箭头和按钮必须绑定真实点击事件,用 navigate/goBack/openModal/showToast/updateState 完成页面跳转、弹层、状态变化或反馈。"

### 坑 9 · 有死按钮或功能没有承接页面

追一句:"请根据模块 2 的 Journey、功能清单和第 8 章整理可点击对象清单,逐一补齐承接。P0 功能必须有页面、弹层或状态变化;P1 占位功能必须给出明确 Toast,不能无响应。"
