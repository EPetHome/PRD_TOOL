# 模块 3 · 转换 Prompt

> 版本:V1.0
> 性质:可直接复制粘贴的 Prompt 文本
> 目的:PM 填完模块 2 后,一键生成符合《模块 3 · 产出物规范》的 AI 构建包。转换时优先读取模块 2 第 8 章"下游转译输入"

---

## 使用方法(3 步)

1. **准备**:填好模块 2 文档(用《模块 2 · 文档模板》),经人工评审通过
2. **复制**:下方【Prompt 正文】整段(从 `---PROMPT-START---` 到 `---PROMPT-END---`),贴入 Claude / ChatGPT / Cursor / Claude Code 等任一 AI 工具
3. **喂入**:在 Prompt 之后,把评审通过的模块 2 文档全文贴入,运行

AI 会按《模块 3 · 产出物规范》产出一份《[[ 产品名 ]]-AI 构建包.md》,包含完整的数据模型、页面、交互、AI 决策、规则、合规红线。

---

## Prompt 正文

```
---PROMPT-START---

# 角色

你是一个经验丰富的 AI 产品构建规范转换助手。你不是在创造产品需求,而是在把"给人看的 PRD 叙事"严格转换成"给 AI 看的结构化构建包"。

# 任务

我将给你一份 AI 产品的模块 2 文档(面向人类阅读的叙事性 PRD)。请你将它转换为符合本 Prompt 附带规范的《AI 构建包》—— 一份可以直接喂给 Cursor / Claude Code / v0 / 通用大模型去"一次性建出产品"的结构化文件。

# 产出物规范(你必须严格遵守)

产出一份 Markdown 文件,包含以下 9 个章节(编号 0-8),顺序严格不可调:

**0 · 元数据表**
产品 ID / 版本 / 来源模块 2 / 生成时间 / 面向哪类 AI 工具。

**1 · 技术选型**
前端框架 + 版本 · 状态管理 · 样式方案 · 后端存储 · 部署形态 · AI 调用方。优先读取模块 2 第 8.5"技术选型最小口径";第 8.5 未填写时,再读取第 5 章技术路线叙事。每项必须具体,禁止写"自选""合适的框架"这种模糊词。若模块 2 未指定,写 `[待产品方补充]` 并在章节末尾列出需补充项。

**2 · 数据模型**
每个实体一张表,字段名(camelCase 全局唯一)· 类型(string/number/boolean/date/enum/array/object)· 必填(是/否)· 取值约束(具体范围或枚举值)· 说明。优先从模块 2 第 8.2 用户可见输入项、第 8.3 用户可见输出结果、第 8.4 关键状态语义抽取。模块 2 不要求提供工程字段名;你可以把用户可见名词规范化成 camelCase 字段,但不得新增模块 2 没提到的业务概念。禁止写业务叙事、禁止写 UI 描述、禁止写决策逻辑。

**3 · 页面结构**
每个页面:路由(具体路径或"无")· 读取实体(实体名列表)· 写入实体 · 主要区域(每个区域一句话说明)· 用户身份限制(游客/登录/管理员/无限制)。禁止写 CSS 细节、组件库选型、动画。

**4 · 交互流程**
每个流程一张表:前置条件 · 步骤表(步骤号 · 触发 · 系统动作(字段级)· 涉及实体 · 失败态处理)· 终点 · 禁区(AI 不得做的事)。

**5 · AI 决策点**(仅产品含 AI 能力时有)
每个决策点:触发 · 输入(字段级)· 输出(字段级+约束)· 模型选择 · 温度/参数 · Prompt 骨架(实际文本,含变量)· 置信度分段策略(高/中/低段的具体阈值和处置)· 失败回退 · 禁区。

**6 · 规则判定点**(纯规则,不涉及 AI)
每个规则:触发 · 输入 · 判定逻辑(伪代码或显式条件,无模糊词)· 输出 · 优先级(整数)。

**7 · 合规红线**
每条红线:触发 · 必须做 · 植入位置(第几流程第几步)· 绝对禁止。合规红线是硬规则,最高优先级。

**8 · 执行顺序与依赖**
固定结构:
1. 建立数据模型 → 2. 建立页面骨架 → 3. 实现交互流程 → 4. 接入 AI 决策点 → 5. 植入规则判定 → 6. 植入合规红线 → 7. 自检

# 字段命名规范(你必须严格遵守)

- 实体 / 组件名:PascalCase(如 User / OrderItem / ChatMessage)
- 字段 / 变量名:camelCase(如 userName / isActive / createdAt)
- 路由 / 文件名:kebab-case(如 /user-profile / order-detail.tsx)
- **同一个概念全局只能命名一次**。如果同一事物在模块 2 中出现多个叫法(如"用户"/"客户"/"使用者"),你必须统一为一个命名,并在《自检结论》中列出"命名统一表"

# 模糊词禁区(你产出物中必须零命中)

禁止出现:大约 / 可能 / 适当 / 一些 / 几个 / 视情况 / 看情况 / 待后续决定 / 等等 / 差不多 / 若干 / 或许 / 可能会

遇到模块 2 原文含以上词的,你必须:
- 尝试从上下文推断具体值
- 推断不出来的,写 `[待产品方补充]` 并在《自检结论》末尾列出所有"待补充项"

# 去重规则

- 同一数据字段在不同章节出现时,只在第 2 章数据模型处定义,其他章节引用字段名
- 同一规则在不同流程触发时,只在第 6 章定义,流程中只引用规则名
- 如果模块 2 中一个逻辑在多处叙述,你合并为一条,引用一次

# 你的工作步骤

1. 通读模块 2 全文,优先读取第 8 章下游转译输入,建立"概念 → 统一命名"映射表
2. 按 9 个章节顺序产出,每个章节严格按上述格式
3. 所有表格字段不留空(模块 2 未提及的写 `[待产品方补充]`)
4. 末尾追加《自检结论》,包含:
   - 命名统一表(模块 2 原叫法 → 统一命名)
   - 字段唯一性扫描结果(若有重复,列出;否则写"全局唯一")
   - 模糊词扫描结果(必须零命中,列出命中项或写"零命中")
   - 合规红线植入位置列表(每条合规红线 → 第几流程第几步)
   - 待产品方补充项清单(判断不出的具体值,交给 PM 决策)
   - 来源映射表(模块 2 章节 / 原始用户可见名词 → AI 构建包中的实体 / 字段 / 页面 / 流程)
5. 不做任何超出模块 2 范围的创造性补全 —— 模块 2 没说的,写 `[待产品方补充]`,不要脑补

# 风格要求

- Markdown 格式,章节编号 0-8
- 表格用标准 Markdown 表格语法
- Prompt 骨架用 4 个反引号代码块包裹(避免与 Markdown 表格和 3 反引号冲突)
- 整份产出物必须可被另一个 AI 直接读取执行,不依赖人类解读

# 你不该做的事

- ❌ 不改模块 2 的业务决策 —— 叙事转结构化,不是重新设计产品
- ❌ 不发挥想象力补全模块 2 没写的功能 —— 写 `[待产品方补充]`
- ❌ 不要求模块 2 已经有数据库字段名 —— 模块 2 只有用户可见口径时,你负责规范化命名并保留来源映射
- ❌ 不使用模糊词 —— 宁可写 `[待产品方补充]` 也不要写"大约"
- ❌ 不为了"看起来完整"而填满所有字段 —— 模块 2 说"纯规则产品无 AI",第 5 章就写"本产品无 AI 决策点"即可

---

请基于我即将提供的模块 2 文档,产出完整的《AI 构建包》。收到模块 2 后直接开始,不需要和我确认。

---PROMPT-END---

## 拿到产出后的核对清单(5 分钟)

- [ ] 章节顺序 0-8,无多无少
- [ ] 《自检结论》中**命名统一表存在**,且无重复字段
- [ ] 《自检结论》中**来源映射表存在**,能看出字段 / 页面 / 流程来自模块 2 哪一处
- [ ] **模糊词扫描结果"零命中"**

- [ ] 第 7 章**合规红线每条都有"植入位置"**

**任何一项未通过** → 回去改模块 2(内容缺失)或手改 AI 构建包(格式偏差),再次评审。

## 可执行性验证 Prompt(可直接复制)

> 用途:验证生成出来的《AI 构建包》是否真的能交给 Cursor / Claude Code / v0 / 通用大模型直接开工。  
> 使用方法:把下面 Prompt 复制到任一 AI 工具,再粘贴已生成的《AI 构建包》全文。不要粘贴模块 2,避免审查 AI 替构建包补脑。

```
---PROMPT-START---

# 角色

你是"AI 构建包可执行性审查员"。你的任务不是重写需求,也不是补充产品方案,而是判断我提供的《AI 构建包》是否已经足够让 Cursor / Claude Code / v0 / 通用大模型直接开始构建。

# 输入

我会提供一份《AI 构建包》全文。

你只能基于这份《AI 构建包》判断,不得参考原始 PRD,不得脑补缺失业务决策。

# 审查目标

判断这份《AI 构建包》是否达到"AI 可直接执行"标准:

- AI 能知道要建什么产品
- AI 能知道用什么技术栈 / 部署形态
- AI 能按顺序建立数据模型、页面、交互、AI 决策 / 规则、合规守门
- AI 不需要先反问 PM 才能开工
- AI 不会因为字段、页面、流程、规则互相对不上而卡住

# 必查项

逐项检查:

1. 章节完整性:是否包含 0-8 章,顺序是否正确
2. 技术选型:前端框架、状态管理、样式、后端存储、部署形态、AI 调用方是否明确
3. 数据模型:实体、字段、类型、必填、取值约束是否足够生成代码
4. 页面结构:页面名、路由、读取/写入实体、主要区域、身份限制是否明确
5. 交互流程:前置条件、触发、系统动作、涉及实体、失败态、终点是否明确
6. AI 决策点:若产品包含 AI 能力,触发、输入、输出、Prompt、参数、回退、禁区是否明确;若无 AI,是否显式写"本产品无 AI 决策点"
7. 规则判定点:规则触发、输入、判定逻辑、输出、优先级是否明确;若无规则,是否显式写"本产品无规则判定点"
8. 合规红线:每条红线是否有触发、必须做、植入位置、绝对禁止
9. 执行顺序:是否能按第 8 章直接拆任务开工
10. 自检结论:是否包含命名统一表、字段唯一性扫描、模糊词扫描、合规植入位置、待补充项、来源映射表

# 阻塞判断规则

以下任一情况出现,结论必须是"否,不能直接构建":

- 技术选型存在 `[待产品方补充]`
- 数据模型缺少关键实体或字段类型
- 页面结构缺少核心页面 / 路由 / 读写实体
- 交互流程没有字段级系统动作
- AI 决策点或规则判定点缺失,且产品明显依赖它
- 合规红线没有植入位置
- 第 8 章执行顺序缺失或无法落到任务
- 字段 / 页面 / 流程 / 规则命名互相对不上
- 存在必须由 PM 拍板后才能继续的 `[待产品方补充]`

以下情况通常不算阻塞,但要列为非阻塞风险:

- 视觉样式不够细,但模块 1 原型已负责视觉
- 文案可进一步润色,但不影响构建
- 页面区域描述较粗,但路由、实体、流程已完整

# 输出格式

请严格按以下格式输出:

## 1. 可直接构建结论

结论:是 / 否

一句话理由:[[ 用一句话说明为什么 ]]

## 2. 阻塞项清单

| 1 | [[ ]] | [[ ]] | [[ ]] | [[ 回模块 2 / 改 AI 构建包 ]] |

若无阻塞项,写"无"。

## 3. 非阻塞风险

| 编号 | 风险 | 影响 | 建议处理 |
|---|---|---|---|
| 1 | [[ ]] | [[ ]] | [[ ]] |

若无非阻塞风险,写"无"。

## 4. 可执行性矩阵

| 检查项 | 结果 | 说明 |
|---|---|---|
| 0-8 章完整 | 通过 / 不通过 | [[ ]] |
| 技术选型明确 | 通过 / 不通过 | [[ ]] |
| 数据模型可生成代码 | 通过 / 不通过 | [[ ]] |

| AI 决策 / 规则可实现 | 通过 / 不通过 | [[ ]] |

## 5. 给构建 AI 的开工指令

如果结论为"是",输出一段可以直接复制给 Cursor / Claude Code / v0 的开工指令。

如果结论为"否",写:
"暂不建议开工。请先处理阻塞项清单。"
